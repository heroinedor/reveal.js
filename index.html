Step 0 : Récupération des sources et installation en local

Rendez-vous au Step 0

Problèmes rencontrés :

  • lecture d'une doc technique fastidieuse et indigeste
  • copie manuelle de librairies de code
  • Risques d'erreurs
  • Temps d'installation trop long

Temps imparti : 5-10 min max

Rapide et simple dans notre cas petit projet

Mais peut être beaucoup plus long sur des projets complexes :

  • 1 à 2 j dans certains cas
  • voire une semaine !
  • à multiplier par le nombre de développeurs

Step 1 : automatisation de l'installation via un outil de build (Webpack)

=> Rendez-vous au Step 1

Solution qui permet :

  • installation en 1 seule commande
  • évite la copie manuelle des dépendances
  • pas d'erreurs grâce à l'automatisation
  • processus rapide standardisé

penser à installer les bon loaders qui vont bien et à configurer directement webpack-dev-server

En général les outils de build sont spécialisés dans un langage

  • JavaScript : Npm, Grunt, Gulp, Webpack, etc.
  • Java : Maven, Gradle, Ivy, Ant, etc.

Step 2 : ajout d'un gestionnaire de configuration (Git)

Problèmatique : Comment préparer un travail en équipe à plusieurs personnes sur les même fichiers ?

Rendez-vous au Step 2

Solution : gestionnaire de configuration qui permet :

  • modification concurrente des fichiers + historisation
  • fusion facilitée des fichiers (surtout Git)
  • développement de feature dans des branches parallèles

Temps imparti : 5-10 min max

Il existe d'autres solutions que Git, mais moins pratiques

Ces solutions sont généralement agnostiques

Musée des dinausores

  • SVN
  • Serena Dimension (à fuir)
  • CVS, Starteam, Clearcase

Signaler les modèles de branching type Git-flow

Step 3 : configuration d'un gestionnaire de dépôt privé (Sinopia)

Problèmatique : Où stocker les dépendances utilisées et celles livrées ?

Rendez-vous au Step 3

Solution : gestionnaire de dépôt privé qui permet :

  • soulager la connexion internet de l'entreprise
  • stocker en interne les dépendances non publiques
  • contrôler la qualité des dépendances

Temps imparti : 5-10 min max

Les méthodes manuelles (copie de fichier sur des serveurs partagés) rencontrent vite leurs limites

  • Méthode Lib dans les gestions de conf : pas une bonne pratique car rend les projets obèses et pose des problèmes lors des montées de version.
  • Partager les libs sur un même lecteur réseau est peu pratique car on peut supprimer une lib partagée par erreur et c’est pas pratique quand on doit en rajouter une nouvelle.

Attention : Sinopia utilisé uniquement à but pédagogique. Solution en fin de vie

Diverses solutions en fonction des langages

  • Java: Nexus, Artifactory, JFrog …
  • JavaScript : Sinopia, Nexus (NPM), private bower...

Step 4 : tests automatiques unitaires et end to end

Problèmatique : comment tester rapidement son code, de manière exhaustive ?

Rendez-vous au Step 4

Solution : automatisation des tests unitaires et e2e

  • augmente la qualité du livrable
  • sécurise les livraisons
  • attention à la maintenance des tests !

Temps imparti : 30 min car assez long

Les tests unitaires doivent rentrer dans les bonnes pratiques car permettent de détecter les erreurs rapidement

Les tests e2e sont la plupart du temps négligés alors qu'ils sont les meilleurs en terme ROI

Attention à la mauvaise pratique du skip des tests

Beaucoup de types de tests différents, plus ou moins automatisables.

Énormément de produits disponibles en fonction du langage et des technos utilisées :

  • Tests unitaires avec tous les XUnit : JUnit, JSUnit, PHPUnit, etc.
  • Tests d’intégration : TestNG, Mockito, Arquilian, Selenium, etc.
  • Tests fonctionnels/ d’Interface / e2e ?? : Selenium, QuickTestPro, Fitnesse, protractor ?
  • Tests de charge : JMeter, neoload, gatling, …

=> Le sujet des tests pourrait faire l’objet d’une présentation à lui tout seul.

Step 5 : Qualité de code

Problèmatique : mesurer et contrôler la qualité du code

Rendez-vous au Step 5

Solution : contrôle automatique du code et reporting

  • évaluation quantitative de la dette technique
  • couverture de code par les tests
  • anticiper les problèmes (NPE, undefined, ...)

Temps imparti : 5-10 min max

par manque de temps, pas mis en place Istanbul

Compatibilité de JavaScript avec SonarCube ?

Peut donner lieu à des dérives et du flicage

Énormément de produits disponibles en fonction du langage et des but recherchés :

  • JavaScript : ESLint, TSLint, Istanbul, …..
  • Java : Sonar, PMD, FindBug, etc....

=> Le sujet de la qualité du code pourrait faire l’objet d’une présentation à lui tout seul.

Step 6 : Chaîne d'Intégration Continue (TravisCI)

Problèmatique : Accélerer et systématiser les tâches précédentes

Rendez-vous au Step 5

Solution : Chaîne d'Intégration Continue

  • éviter le syndrôme du "ça marchait sur mon poste"
  • détecter les erreurs à chaque push
  • gain de temps en développement

Temps imparti : 10-20 min car peut être long

Dans notre cas, un seul environnement, mais plusieurs en entreprise

Solutions dépendent du langage utilisé et de la politique d’entreprise choisie :

  • Internes à l’entreprise (installation sur un serveur à part entière) : Jenkins, Team Foundation Server, GitLabCI …
  • Externalisé : TravisCI, Atlassian Bamboo, ...

=> Le sujet de la qualité du code pourrait faire l’objet d’une présentation à lui tout seul.
